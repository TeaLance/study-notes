Chapter 4

The Processor 處理器

有了會算數的 ALU，接著要把記憶體、暫存器、ALU 接成能自動執行指令的處理器。 先學最直覺的「單週期」設計，再用「管線」把效能拉高——而管線帶來的麻煩（危障）正是本章重頭戲。

4.1指令執行的五大步驟

不論哪種指令，MIPS 都可拆成五個階段，這也正是後面五級管線的由來：

  • IF (Instruction Fetch)：用 PC 從指令記憶體取指令，PC ← PC + 4。
  • ID (Instruction Decode)：解碼，讀暫存器檔的兩個來源暫存器，符號延伸 immediate。
  • EX (Execute)：ALU 運算——算結果 / 算記憶體位址 / 算分支目標。
  • MEM (Memory access)：只有 lw / sw 會存取資料記憶體。
  • WB (Write Back)：把結果寫回暫存器檔。
三類指令：算術邏輯 add, sub, and, or, slt；記憶體 lw, sw；控制轉移 beq, j。它們走的路徑不同，但共用同一組硬體。

4.3控制訊號 Control Signals

控制單元 (control unit) 解碼指令的 opcode，產生上面那排控制線去「指揮」MUX、記憶體與暫存器讀寫。常見訊號：

  • RegDst：寫回的暫存器編號來自 rt(0) 還是 rd(1)。
  • ALUSrc：ALU 第二運算元來自暫存器(0) 還是符號延伸的 immediate(1)。
  • MemRead / MemWrite：是否讀 / 寫資料記憶體。
  • MemtoReg：寫回的值來自 ALU(0) 還是資料記憶體(1)。
  • RegWrite：是否寫回暫存器；Branch：是否為分支（搭配 Zero 決定是否跳）。
  • ALUOp：兩位元，配合 funct 欄位由 ALU control 產生真正的 4-bit ALU 控制。

4.4管線化 Pipelining

單週期的缺點：時脈被最慢的指令（lw）綁死。管線 (pipeline) 像洗衣店流水線—— 把指令切成五級，讓不同指令的不同階段同時進行。它不縮短單一指令的時間，而是大幅提高吞吐量 (throughput)。

理想加速比 ≈ 管線級數（這裡 = 5）
條件：各級時間均衡、且沒有危障 (hazard) 造成停頓
  • 級與級之間插入管線暫存器（IF/ID、ID/EX、EX/MEM、MEM/WB）保存中間狀態。
  • 穩定狀態下每個時脈完成一條指令（CPI → 1），但啟動有「填管線」的延遲。

4.5危障 Hazards 與前送 Forwarding

管線重疊執行會產生危障 (hazard)——三種：

  • 結構危障 (structural)：硬體資源不夠用（例如同一個記憶體同時要取指令又要讀資料）。解法：指令/資料記憶體分開。
  • 資料危障 (data)：後面指令要用前面還沒寫回的結果。解法：前送 (forwarding) 把 ALU 結果直接送回去；lw 之後仍需 1 個 停頓 (stall / bubble)。
  • 控制危障 (control)：分支要等條件算完才知道跳不跳。解法：分支預測 (branch prediction)、提早判斷分支。

Chapter 5

Memory Hierarchy 記憶體階層

處理器越來越快，記憶體卻沒跟上（CPU-DRAM gap）。解法不是做一個又快又大又便宜的記憶體（不存在）， 而是疊出一個階層：用小而快的擋在前面，大而慢的放後面，靠局部性讓多數存取都打中快的那層。

5.1局部性原理 Principle of Locality

程式的記憶體存取不是隨機的，而有局部性 (locality)——這是整個記憶體階層能成立的關鍵：

  • 時間局部性 (temporal)：剛用過的東西，很可能馬上又用（迴圈變數、指令）。
  • 空間局部性 (spatial)：用了某位址，附近的位址也很可能被用（陣列、循序指令）。

因此把資料以區塊 (block / line) 為單位，從慢的層搬到快的層，就能用少量快記憶體服務大多數存取。

5.2記憶體技術 SRAM / DRAM / Disk

技術原理速度每 GB 成本用途
SRAM正反器（6 顆電晶體/bit），不需更新最快 (~1ns)最貴Cache
DRAM電容存電荷（1T/bit），需週期性 refresh中 (~50ns)中主記憶體
Flash / Disk非揮發；磁碟有機械尋軌最慢 (µs~ms)最便宜儲存
  • 理想記憶體：SRAM 的速度 + Disk 的容量與成本——做不到，所以才需要階層。
  • DRAM 以列 (row) 為單位讀取整列；DDR / QDR 在時脈邊緣多次傳資料來提高頻寬。

5.3Cache 基礎與對映方式

Cache 是 CPU 與主記憶體之間的小型 SRAM。一次存取若在 cache 找到叫 命中 (hit)，找不到叫 失誤 (miss)，要從下層搬一個區塊上來（付出 miss penalty）。

hit ratio = hits / accesses　　miss ratio = 1 − hit ratio
平均存取時間 AMAT = hit time + miss ratio × miss penalty

位址被切成三段來定位：

［　Tag　｜　Index（選哪一行）｜　Offset（區塊內位元組）　］
  • 直接對映 (direct-mapped)：每個記憶體區塊只能放在固定一行（index = 區塊號 mod 行數）。簡單但易衝突。
  • 組相聯 (set-associative)：每組有 N 行，區塊可放組內任一行，衝突變少。
  • 全相聯 (fully-associative)：可放任意行，最少衝突但比對成本最高。
  • 寫入策略：write-through（同時寫下層）vs write-back（只寫 cache，靠 dirty bit 標記，換出時才寫回）；valid bit 標記該行是否有效。

5.4虛擬記憶體 Virtual Memory 與 TLB

虛擬記憶體 把「主記憶體當成磁碟的 cache」：每個程式以為自己擁有完整連續的位址空間， 硬體 + OS 再把虛擬位址翻成實體位址。單位是 page（通常 4KB）。

  • 頁表 (page table)：記錄每個虛擬頁對應的實體頁框；找不到（不在記憶體）叫 page fault，要從磁碟搬入。
  • TLB (Translation Lookaside Buffer)：頁表的 cache。每次存取要先翻譯位址，若每次都查頁表（在記憶體）太慢，TLB 用小型快取存最近的翻譯。
  • 對應關係：cache 的 hit/miss ↔ TLB hit/miss ↔ page fault，都是「上層找不到就往下層拿」的同一個思想。
一條 load 的完整路徑：虛擬位址 → 查 TLB（翻成實體位址）→ 查 cache → 命中就拿資料，失誤就往主記憶體 / 磁碟一路追下去。整個 Chapter 5 就是這條鏈上每一環的最佳化。

Q2set 數量 · 三欄位 Tag / Index / Offset · total cache size

Q2 題目：給定 cache 容量、block 大小與相聯度，求 ① 快取的 set 數量、② 位址三欄位 Tag / Index / Offset 各幾位元、③ set-associative 下的 total cache size（含標記與 valid bit 的實際儲存量）。

解法：先記住三條關係

一個位元組位址 (byte address) 被切成三段來定位資料：

［　Tag（比對是不是這個區塊）　｜　Index（選哪一個 set）　｜　Offset（block 內第幾個位元組）　］
Offset 位元 = log2(block 大小 bytes)
#blocks = cache 資料容量 ÷ block 大小　　#sets = #blocks ÷ 相聯度 (associativity)
Index 位元 = log2(#sets)　　Tag 位元 = 位址寬度 − Index − Offset
  • set 數量是關鍵：DM（1-way）時 #sets = #blocks；2-way 時 #sets = #blocks ÷ 2；全相聯時 #sets = 1（Index = 0 位元）。
  • 用哪個 set：(block number) mod (#sets)。相聯度越高、#sets 越少 → Index 變短、Tag 變長。
  • total cache size（實際儲存量）不是只有資料，每一行還要存 Tag 與 1 個 valid bit：
total size = #sets × 相聯度 × (　block 資料位元 + Tag 位元 + 1 (valid)　)

講義例題 A（投影片 p.29 · Address Subdivision）

32-bit 位址、1 word/block（= 4 bytes）、direct mapped、1024 個 block。求 tag 大小與 cache 總位元數。

步驟計算結果
Offset 位元log₂(4 bytes)2
#sets（DM）= #blocks = 10241024
Index 位元log₂(1024)10
Tag 位元32 − 10 − 220
每行位元data 32 + tag 20 + valid 153
總位元數1024 × 5354,272 bits

→ 資料本體只有 1024×32 = 32,768 bits（4 KB），但加上 Tag 與 valid bit 後實際要存 54,272 bits，這就是標記額外負擔 (overhead)。

講義例題 B（投影片 p.37 · Mapping an Address）

cache 有 64 個 block、block 大小 16 bytes。問位元組位址 1200 對映到哪個 block？三欄位各幾位元（32-bit 位址）？

block address = ⌊1200 ÷ 16⌋ = 75　→　block number = 75 mod 64 = 11
Offset = log₂16 = 4　Index = log₂64 = 6　Tag = 32 − 6 − 4 = 22

Q3DM 與 2-way 的速度比較（依講義）

Q3 題目：同樣容量下，比較直接對映 (DM, 1-way) 與 2-way set associative 哪個比較快。

解法：速度要看 AMAT，不能只看 miss rate（依講義）

  • 2:1 Cache Rule（講義 p.56）：容量 N 的 DM，其 miss rate ≈ 容量 N/2 的 2-way。也就是「2-way 用一半容量就追平 DM 的失誤率」——2-way 失誤率較低。
  • 但「執行時間才是最終標準」：提高相聯度要先選 set 再用多工器從多個 way 挑出資料，hit time / 時脈週期 (CCT) 會變長。
  • Hill [1988]：2-way 相對 1-way 的 hit time，外部 cache +10%、內部 cache +2%。
AMAT（平均存取時間）= hit time + miss rate × miss penalty
DM：　 AMAT = hitbase + missDM × penalty
2-way：AMAT = hitbase × CCT 係數(≈1.10) + miss2-way × penalty

所以結論是取捨：2-way 把 miss rate 壓低、卻把 hit time 拉高。哪個快要把兩者代進 AMAT 才知道。

講義 AMAT 對照表（p.57，cycles，CCT：2-way ×1.10）

綠＝該容量下 2-way 較快；紅＝2-way 反而較慢（hit time 變長的代價蓋過 miss rate 的改善）。

結論：cache 容量小時（≤ 4 KB），衝突失誤多，2-way 降 miss rate 的好處大於 hit time 的代價 → 2-way 較快；容量大到一定程度（≥ 8 KB）後 miss rate 本來就低，2-way 多出來的 hit time 反而拖累整體 → DM 較快。這正是講義說「相聯度提高，AMAT 不一定改善」的意思。
